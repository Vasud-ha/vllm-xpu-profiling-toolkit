VTune + vLLM Conditional Profiling on Intel Xe

ROI-gated GPU-Hotspots profiling for vLLM serving on Intel Xe / Xe2 (BMG, PVC) — start VTune paused, drive collection via vtune -command or ITT __itt_resume/__itt_pause, capture only the inference window.
Intel VTune Profiler 2024.x / 2025.x vLLM v1 (≥ 0.10) Intel Xe — BMG / PVC GPU-Hotspots -start-paused + ITT resume/pause

Authoritative sources: Download — Intel VTune Profiler download · Docs — VTune Profiler documentation hub · User Guide · GPU-Hotspots analysis · ITT API reference

1. Summary

Profiling a full vLLM stack with VTune in default mode produces 10–50 GB of trace data dominated by Python startup, model load, JIT compilation, tokenizer overhead and warmup. The steady-state inference kernels — the actual subject of analysis — are buried in noise.

This skill defines a reproducible conditional / ROI profiling flow that:

  • Starts VTune paused via -start-paused, so model load and JIT compiles are excluded.
  • Drives collection via external vtune -command resume / pause / stop or in-process ITT API (__itt_resume / __itt_pause).
  • Patches vllm.v1.worker.gpu_worker.Worker.execute_model so the ROI sits at the tightest correct boundary — pure GPU compute, no HTTP/asyncio/tokenizer leakage.
  • Synchronizes the XPU queue (torch.xpu.synchronize()) before pause to guarantee every kernel submitted inside the ROI is captured.
Quantitative outcome
Result size drops from 10–50 GB to 100 MB – 2 GB; kernel attribution becomes high; warmup contamination is eliminated; GUI analysis time falls from minutes to seconds.

Validated on gnrsp-bmg3.iind.intel.com against:

  • intel/vllm:0.14.1-xpu (vllm 0.1.dev14456+gde3f7fe65) — original validation environment.
  • intel/vllm:0.17.0-xpu — re-validated in container unitrace_vllm_test_vasu; the ROI flow, ITT symbol resolution, and VTune result structure behave identically. No script changes required.
  • intel/vllm:0.21.0-ubuntu24.04 — re-validated in container skill-test-vasu; see verification_log_vllm_0.21.md in the toolkit repo. VTune 2025.x is preflight-blocked on BMG (libigdmd.so gap); use VTune 2026.0 or switch to unitrace for BMG kernel attribution.

Validation base: oneAPI 2025.3, kernel 6.17, Intel Graphics [0xe223] (BMG / Xe2), VTune 2025.x / 2026.0.

2.2 ROI control contract

benchmark script (curl / OpenAI SDK / vllm bench serve) | | vtune -command resume (or __itt_resume from inside Worker.execute_model) v VTune driver ──► starts collecting GPU-Hotspots samples + L0 API timeline | | ... inference traffic ... | | vtune -command pause (or __itt_pause from inside Worker.execute_model) v VTune driver ──► stops collecting; result remains open | | vtune -command stop v VTune driver ──► finalize result-dir (resolve symbols, write index)

2.3 Why each control step is needed

StepReason
-start-paused at execModel load + JIT generates 10–30 s of GPU activity that is NOT inference and dominates hotspot rankings.
Process tree under one vtune drivervLLM v1 splits the engine across processes. Only descendants of the vtune driver are reachable by vtune -command resume. Always launch via vtune -- python …, never attach after the fact.
Warmup remains pausedFirst runs through any new shape compile SYCL/IPEX kernels; those compile events crowd out steady-state hotspots.
torch.xpu.synchronize() before pauseXPU submissions return immediately; if pause fires before kernels finish, VTune misses the actual GPU work. Most common cause of "empty trace".
-command stopPause leaves the result open. Stop finalizes — writes index, resolves symbols, closes file handles.
External command vs ITT — when to use which
External (vtune -command resume/pause) — process-tree-wide, no code changes, perfect for benchmark drivers. ITT (__itt_resume / __itt_pause) — per-process; in vLLM v1 you must call it from inside the EngineCore/Worker subprocess, not the API parent. Use it for step-level task tagging or when you can't run an external driver.

5. ITT + vLLM Integration

5.1 ITT API surface

Control:
  __itt_resume()          // resume ALL collection (process-global)
  __itt_pause()           // pause ALL collection

Task / domain (timeline annotations):
  __itt_domain_create()
  __itt_task_begin()
  __itt_task_end()

Full reference: ITT API documentation.

5.2 ctypes wrapper (zero deps)

import ctypes, ctypes.util, os, contextlib

_SEARCH = [
  "/opt/intel/oneapi/vtune/latest/lib64/libittnotify.so",
  "/opt/intel/vtune_profiler/lib64/libittnotify.so",
  os.path.join(os.environ.get("VTUNE_PROFILER_DIR", ""), "lib64/libittnotify.so"),
]
def _load():
    n = ctypes.util.find_library("ittnotify")
    if n:
        try: return ctypes.CDLL(n)
        except OSError: pass
    for p in _SEARCH:
        if p and os.path.exists(p):
            try: return ctypes.CDLL(p)
            except OSError: continue
    return None

_lib = _load()
_resume = getattr(_lib, "__itt_resume", None) if _lib else None
_pause  = getattr(_lib, "__itt_pause",  None) if _lib else None

def itt_available(): return _resume is not None
def itt_resume():    _resume and _resume()
def itt_pause():     _pause  and _pause()

@contextlib.contextmanager
def vtune_roi():
    import torch
    itt_resume()
    try: yield
    finally:
        if torch.xpu.is_available(): torch.xpu.synchronize()
        itt_pause()

5.4 Where to put the ROI

HTTP POST /v1/completions
  |  [API parent process]
AsyncLLM.generate()                       (asyncio)
  |  [IPC: ZMQ → EngineCore subprocess]
EngineCore.step()                          (scheduler + KV cache)
  |
  +-- model_executor.execute_model(scheduler_output)
        |  [GPU worker process(es): one per TP rank]
      Worker.execute_model(scheduler_output)   ◄── BEST ROI BOUNDARY
        |
        +-- model.forward(...)                  ◄── tightest boundary (per-step)

5.5 Monkey-patch sketch

import functools, torch
from vllm.v1.worker.gpu_worker import Worker
from vtune_itt import itt_resume, itt_pause

_orig_exec = Worker.execute_model

@functools.wraps(_orig_exec)
def _exec_with_roi(self, scheduler_output, *args, **kwargs):
    itt_resume()
    try:
        out = _orig_exec(self, scheduler_output, *args, **kwargs)
    finally:
        torch.xpu.synchronize()
        itt_pause()
    return out

Worker.execute_model = _exec_with_roi

Phase tags (prefill / decode) can be layered on top using __itt_task_begin/end with the scheduler_output's scheduled_new_reqs / scheduled_cached_reqs to classify.

5.6 ITT overhead

CallVTune attachedVTune not running
__itt_resume / __itt_pause (C)50–200 ns2–5 ns
ittapi.resume() (Python pkg)300–800 ns10–20 ns
ctypes wrapper500 ns – 2 µs20–50 ns
torch.xpu.synchronize()1–50 µs (queue-depth dependent)N/A

For LLM inference (forward = 10–500 ms), per-step ITT + sync overhead is < 0.5 %.

6. Scripts

Two files comprise the production wrapper (mirrors the unitrace skill layout):

  • scripts/serve_with_vtune.py — vLLM server wrapper; installs the Worker.execute_model ROI monkey-patch.
  • scripts/run_vtune_vllm.sh — driver: vtune -collect gpu-hotspots -start-paused + benchmark loop bracketed by vtune -command resume/pause/stop.

6.1 run_vtune_vllm.sh skeleton

#!/usr/bin/env bash
set -euo pipefail
source /opt/intel/oneapi/setvars.sh --force

MODEL=${MODEL:-meta-llama/Llama-3.1-8B-Instruct}
PORT=${PORT:-8000}
PHASE=${VTUNE_PHASE:-decode}            # prefill | decode | mixed
INPUT_LEN=${INPUT_LEN:-512}
OUTPUT_LEN=${OUTPUT_LEN:-256}
NUM_PROMPTS=${NUM_PROMPTS:-20}

STAMP=$(date +%Y%m%d_%H%M%S)
RESULT=$(realpath ./vtune_results/${STAMP}_${PHASE})
mkdir -p "$RESULT"

# 1. Launch vLLM under VTune, paused
vtune -collect gpu-hotspots -start-paused -result-dir "$RESULT" \
  -knob characterization-mode=overview \
  -knob collect-programming-api=true \
  -- python -m vllm.entrypoints.openai.api_server \
       --model "$MODEL" --port "$PORT" --dtype float16 --enforce-eager \
       --max-model-len 4352 --max-num-batched-tokens 4352 &
SERVER_PID=$!

# 2. Wait for /v1/models
for i in $(seq 1 600); do
  curl -sf http://localhost:$PORT/v1/models >/dev/null 2>&1 && break
  sleep 1
done

# 3. Bracket the ROI
vtune -command resume -r "$RESULT"
vllm bench serve --base-url http://localhost:$PORT --model "$MODEL" \
     --num-prompts "$NUM_PROMPTS" --max-concurrency 1 \
     --random-input-len "$INPUT_LEN" --random-output-len "$OUTPUT_LEN" \
   | tee "$RESULT/bench.log"
vtune -command pause -r "$RESULT"

# 4. Shut down vLLM, finalize VTune
kill -INT $SERVER_PID
wait $SERVER_PID 2>/dev/null || true
vtune -command stop -r "$RESULT"

# 5. Headless reports
vtune -report summary  -r "$RESULT" -format csv  > "$RESULT/summary.csv"
vtune -report hotspots -r "$RESULT" -group-by computing-task -format csv -limit 25 \
                                                  > "$RESULT/hotspots.csv"

echo "result: $RESULT"

7.4 BMG-specific notes

  • Decode is BW-bound — expect 10–40 % EU utilization. Correct, not a profiling failure.
  • Prefill is compute-bound — expect 50–80 % EU utilization for long sequences.
  • --enforce-eager is required on intel/vllm-xpu to avoid the llvm-foreach SEGV in SYCL JIT and to get clearer kernel names.
  • Phase ratio: at BS=1, prefill_GPU_time : decode_GPU_time_per_token ≈ seq_len / 8. A 512-token prompt with 256-token response → decode dominates ≈ 99 % of GPU time. Profile prefill and decode in separate runs for clean attribution.

7.5 Opening the result in the VTune viewer

A finalized result-dir opens in three ways — on the collection host, over SSH, or on your local workstation after copying it off. Pick whichever fits your setup:

On the collection host (terminal)

source /opt/intel/oneapi/setvars.sh                                       # puts vtune-gui on PATH
vtune-gui ./vtune_results/<timestamp>_<phase>                             # opens the result dir in the VTune GUI

On a headless / SSH host

ssh -X <user>@<host>                                                      # or -Y for trusted forwarding
source /opt/intel/oneapi/setvars.sh && vtune-gui ./vtune_results/<timestamp>_<phase>

Locally — copy the dir off first

scp -r <user>@<host>:/path/to/vtune_results/<timestamp>_<phase> ~/vtune_local/   # copy the whole result dir
vtune-gui ~/vtune_local/<timestamp>_<phase>                                   # open it in the local VTune GUI

Locally — with the VTune application (GUI, no terminal)

  1. Launch Intel VTune Profiler from your Start menu / Applications / Launchpad (or run vtune-gui from a terminal that has setvars.sh sourced).
  2. In the VTune window, click Open Result (or File → Open → Result…), navigate to the copied result dir, and select the <timestamp>_<phase>.vtune file inside it — VTune loads the Summary tab automatically.
Copy the whole directory, not individual files
A result dir needs config.cfg, data.0/, log/, and runss/ together. Copying only the .vtune file will fail to open. On the local machine, VTune Profiler must be installed and its major version must be ≥ the collector's (e.g. open a 2025.x result with 2025.x or 2026.0, not 2024.x).

8. Troubleshooting

SymptomRoot causeFix
GPU Time = 0 Collection paused before GPU kernels completed Add torch.xpu.synchronize() immediately before itt_pause() / vtune -command pause
Empty GPU timeline Conflicting tracing layer enabled (PTI / extra L0 layers) unset ZE_ENABLE_TRACING_LAYER ZE_LOADER_LAYERS_ENABLE PTI_ENABLE_COLLECTION PTI_ENABLE_RUNTIME_TRACING
Huge CPU time, tiny GPU fraction ROI too wide — captures HTTP / asyncio / tokenizer Move ROI inward: AsyncLLMEngine.generate → EngineCore.step → Worker.execute_model → model.forward
Result file > 10 GB Too much data captured Set -knob collect-programming-api=false, -knob collect-host-gpu-data-transfers=false, raise -knob gpu-sampling-interval=5, or profile fewer requests
ITT calls do nothing (a) lib not found, (b) -start-paused missing, (c) vLLM launched outside vtune, (d) LD_PRELOAD conflict Set VTUNE_PROFILER_DIR / extend LD_LIBRARY_PATH; always use vtune -- python …; check the wrapper logs ITT available: True in both the API parent and the engine subprocess
vtune -command fails: "no collection" Path mismatch or vLLM crashed at startup Reuse the same $VTUNE_RESULT variable everywhere; check vtune -command status -r $RESULT and ps aux | grep vllm
Unreadable kernel names (<unknown>) AOT-compiled XPU kernels without debug symbols For custom SYCL: icpx -fsycl -gline-tables-only -O2. In GUI: add PyTorch-XPU / IPEX build paths under Configure Analysis → Search Directories

Multi-rank: one rank not profiled ITT is per-process; external command resumes only the vtune-owned tree Patch Worker.execute_model so each rank calls ITT itself; ensure all ranks are descendants of the same vtune -- python … invocation
llvm-foreach SEGV in SYCL JIT (BMG) torch.compile interaction with intel/vllm-xpu Launch vLLM with --enforce-eager

9. Conclusion

Key takeaways
  1. VTune + -start-paused + ROI control ≡ clean trace. Without all three, you're back to a multi-GB whole-lifetime trace.
  2. Launch under VTune, never attach after the fact — vLLM v1's split topology (API parent + EngineCore subprocess + per-rank Workers) requires umbrella ownership of all descendants.
  3. Always sync before pause. torch.xpu.synchronize() is the single most important correctness rule; without it, in-flight kernels are missed.
  4. Always validate post-run: GPU Time > 0, top hotspots are real GPU kernels, ROI duration matches benchmark wall-clock within ±20 %.
  5. Decode that looks under-utilized (10–30 % EU) is not a bug — decode is BW-bound by physics.
  6. Use a hybrid ROI: external vtune -command resume/pause bracketing the whole window; per-step ITT task tags inside it for phase lanes.
