TL;DR — one sentence each

PyTorch profiler
What did my model do?
Hooks the framework. Sees nn.Module.forward, autograd, Python frames, and the kernel each one launches. Best for "which layer is slow".
Intel VTune GPU‑Hotspots
Why is the GPU slow?
Hardware sampler. Reads Xe EU counters, occupancy, memory bandwidth, stall reasons. Best for "is this kernel compute‑ or BW‑bound".
pti‑gpu unitrace
What did the runtime do?
Level Zero / SYCL / oneCCL / oneDNN tracer. Shows every host‑side dispatch and every GPU kernel with timing. Best for "where did the time actually go on the timeline".
None of these is "the best profiler" — they answer different questions. The right move is usually one tool to localize and another to explain.

Where each tool hooks the stack

vLLM on Intel Xe has at least seven layers between your prompt and the GPU. Each profiler hooks at a different layer, which is why their traces look so different.

Read this picture as: PyTorch profiler is shallow but knows your model, unitrace is wide and knows the runtime, VTune is deep and knows the silicon.

Per‑tool detail

1. PyTorch profiler — framework view

PyTorch's built‑in profiler hooks the dispatcher and Python frame entry, then asks the backend (CUDA via CUPTI, or Intel XPU via Kineto + Level Zero hooks) to also stamp the GPU kernel each op launched. vLLM ships first‑class integration: enable the torch backend at server start and POST /start_profile / /stop_profile just work.

The gate changed at vLLM 0.17 — the old VLLM_TORCH_PROFILER_DIR env var is now silently ignored (server logs “Unknown vLLM environment variable”) and the profile router only attaches when --profiler-config supplies a ProfilerConfig.

How to bracket the ROI

# vLLM >= 0.17 — --profiler-config JSON (current)
vllm serve meta-llama/Llama-3.1-8B-Instruct --enforce-eager --max-num-seqs 32 \
  --profiler-config '{"profiler":"torch","torch_profiler_dir":"/abs/path","torch_profiler_use_gzip":true}'

# vLLM <= 0.14 (legacy) — env var still honored
# export VLLM_TORCH_PROFILER_DIR=$HOME/vllm_profile
# vllm serve meta-llama/Llama-3.1-8B-Instruct --enforce-eager

# in another shell — or use `vllm bench serve --profile`
curl -X POST http://localhost:8000/start_profile
<run inference>
curl -X POST http://localhost:8000/stop_profile

What the trace looks like

Open the .pt.trace.json in Perfetto. You see a CPU thread with the call tree (execute_model → LlamaForCausalLM.forward → LlamaDecoderLayer.forward → LlamaAttention.forward → F.scaled_dot_product_attention → aten::matmul) and a GPU/XPU stream with the actual kernel each op launched. ITT/NVTX‑style annotations (prefill, decode) anchor the timeline.

What it cannot tell you

  • EU active % · HBM bandwidth utilization · stall reasons — needs HW counters (VTune).
  • Host‑side L0 dispatch latency · oneCCL allreduce internals · oneDNN primitive selection — needs runtime tracing (unitrace).

2. Intel VTune GPU‑Hotspots — hardware view

VTune samples GPU hardware counters (EU active, EU stall, IPC, occupancy, sub‑slice load, HBM/LLC bandwidth) and ranks kernels by GPU time. With -knob profiling-mode=source-analysis on Xe it can attribute EU stalls down to instructions in a single kernel. -start-paused + ITT API gives you the same ROI control surface as the other tools.

How to bracket the ROI

vtune -collect gpu-hotspots -start-paused \
      -knob collect-programming-api=true \
      -result-dir ./vtune_results/run1 \
      -- python -m vllm.entrypoints.openai.api_server --model ... &

# warmup, then
vtune -command resume -r ./vtune_results/run1
<benchmark>
vtune -command pause  -r ./vtune_results/run1
vtune -command stop   -r ./vtune_results/run1

What the report looks like

In vtune-gui: Summary tab gives GPU Time, EU Array Active %, GPU Memory BW. Bottom‑up tab ranks kernels by GPU time and shows per‑kernel EU stalls. Platform tab shows the L0 queue with kernel submissions. Headless alternative: vtune -report hotspots/summary/hw-events emit CSVs for CI / regression diffing.

What it cannot tell you

  • Which nn.Module emitted that kernel — you only see kernel names. Inject ITT tasks if you need module attribution.
  • Per‑step Python‑side latency between launches — outside the GPU lens.

3. pti‑gpu unitrace — runtime trace view

unitrace is the GPU tracer in intel/pti-gpu. It hooks Level Zero (and optionally OpenCL/SYCL/oneCCL/oneDNN/MPI) at runtime. Output is a Chrome‑trace JSON per process plus an aggregated device‑timing summary at exit (with -d). Same ROI mechanism as VTune (--start-paused + ITT), exposed in this skill via /start_profile / /stop_profile.

How to bracket the ROI

cd ~/.claude/skills/unitrace-vllm-profiling/scripts
bash run_unitrace_vllm.sh   # uses unitrace --start-paused under the hood

curl -X POST http://localhost:9090/start_profile
<inference>
curl -X POST http://localhost:9090/stop_profile

What the trace looks like

Per‑process python.<pid>.json in Perfetto. The engine‑core process shows GPU kernels with full launch metadata (SIMD width, group sizes, shapes inlined into the name). With --chrome-call-logging you also see every zeCommandListAppendLaunchKernel on the host. With --chrome-ccl-logging you see oneCCL allreduce/allgather. With --chrome-itt-logging the /start_profile markers show as ITT regions framing the kernels.

What it cannot tell you

  • EU stalls / BW utilization — no hardware counters. unitrace is a tracer, not a sampler.
  • Module hierarchy — same as VTune: kernel names only unless you push ITT ranges from your code.

Advantages and disadvantages

PyTorch profiler

    Pros
  • Sees the nn.Module tree — layer attribution for free
  • Vendor‑agnostic (CUDA + XPU)
  • First‑class vLLM integration (--profiler-config on ≥0.17, VLLM_TORCH_PROFILER_DIR on ≤0.14, vllm bench serve --profile)
  • Perfetto‑readable output
  • Lowest setup cost
  • Captures Python frames + autograd
    Cons
  • No HW counters (EU%, BW, stalls)
  • Op‑level overhead can skew very short kernels
  • Trace size grows fast with record_shapes / with_stack
  • CUDA‑graph / torch.compile hides individual kernel names
  • Limited insight into oneCCL/oneDNN internals

VTune GPU‑Hotspots

    Pros
  • The only one with EU active %, BW%, and stall breakdowns
  • Source‑analysis mode attributes stalls to instructions
  • Per‑kernel ranking with HW context
  • Headless CSV reports (-report hotspots)
  • Same ROI semantics as unitrace (-start-paused + ITT)
  • L0 timeline with collect-programming-api=true
    Cons
  • Intel only
  • Highest in‑ROI overhead of the three
  • Largest result dirs (can hit 1–2 GB)
  • GUI heavy; headless workflow takes more plumbing
  • Knob names drift across releases
  • ZE env hygiene matters (ZE_ENABLE_TRACING_LAYER must be unset)

pti‑gpu unitrace

    Pros
  • Fine‑grained timeline — every dispatch and kernel
  • Sees oneCCL, oneDNN, MPI, SYCL, OpenCL, L0 in one trace
  • Smallest trace files of the three
  • Kernel names include SIMD width and group sizes
  • Lower in‑ROI overhead than VTune
  • Per‑process Chrome‑trace JSON; merges cleanly in Perfetto
    Cons
  • No hardware counters (timestamps only)
  • Intel / oneAPI only
  • Build step required (BMG/Xe2 needs a generator patch today)
  • No native module attribution — you must emit ITT ranges
  • Less mature than VTune; thinner UI surface

Best‑fit scenarios

"I want to know which decoder layer / which attention head is slow."
You need module attribution. Only PyTorch profiler reads nn.Module natively.
PyTorch profiler
"My GEMM kernel takes 5 ms — is it compute or BW limited?"
You need HW counters: EU active %, GPU memory BW%, stall reasons. Only VTune gives these.
VTune GPU‑Hotspots
"There's a 200 µs gap between every kernel launch — what is the host doing?"
You need the L0 host‑call timeline. unitrace with --chrome-call-logging shows every zeCommandList* call. VTune (with collect-programming-api) also works but heavier.
unitrace · VTune (alt)
"TP‑2 is slower per‑step than TP‑1 — how big is allreduce?"
unitrace has dedicated --chrome-ccl-logging. PyTorch profiler shows them as opaque NCCL/CCL ops. VTune shows the kernels but lumps the collective.
unitrace (preset=ccl)
"I want a regression gate in CI — pass/fail by GPU‑time delta."
VTune has -report hotspots -format csv that diffs cleanly. PyTorch traces also work but you'll write more glue.
VTune · unitrace report (auto‑generated HTML)
"I'm chasing a 2 s gap during prefill — tokenizer? scheduler? GPU?"
PyTorch profiler walks the Python frame, so the gap will show as a Python span between prefill and the next kernel launch. VTune sees only "no GPU activity"; unitrace sees only "no L0 calls".
PyTorch profiler
"Decode busy% is 18% — is that healthy?"
Healthy decode is BW‑bound and 10–40% EU active is normal. You need EU% to confirm. VTune is the answer.
VTune
"I just want to count kernels and sum their GPU time per step."
unitrace's Chrome JSON + the auto‑generated HTML report does exactly this with the smallest trace and least overhead.
unitrace
"Cross‑vendor (CUDA today, XPU tomorrow) consistency in CI."
Only PyTorch profiler is vendor‑agnostic. VTune is Intel; unitrace is Intel.
PyTorch profiler
"I see SEGV / unreadable kernel names in my XPU trace."
Always launch with --enforce-eager on intel/vllm-xpu; torch.compile + SYCL JIT misbehaves there. Applies to all three tools.
launch flag, not a tool choice

unitrace with --chrome-dnn-logging shows the primitive. VTune sees a kernel name; PyTorch profiler sees an aten op.
unitrace (preset=dnn)

All three answer this. Pick by what else you need; unitrace's HTML report surfaces it directly.

Decision flow

Using them together

The three tools share an ROI mechanism (ITT API + paused start), so you can run them back‑to‑back on the same workload and reuse the same client/benchmark script. A typical investigation cascade:

PyTorch profiler → localize to a layer → unitrace → identify the kernel & how often it runs → VTune source‑analysis → explain why it stalls

Tip: drive all three from the same client

The vLLM server can have all three of --profiler-config (torch), ITT‑driven unitrace endpoints, and ITT‑driven VTune endpoints wired up at once. Pick the active backend by what wraps the python ... launch: 

plain                            → PyTorch profiler endpoints active
vtune -collect gpu-hotspots ...  → ITT calls drive VTune (PyTorch endpoints still work, but trace is owned by VTune)
unitrace --start-paused ...      → ITT calls drive unitrace

The client — curl /start_profile && vllm bench serve ... && curl /stop_profile — is identical in all three cases.

Tip: emit ITT tasks for module hierarchy in unitrace and VTune

Register forward‑pre/post hooks on your decoder layers and call __itt_task_begin / __itt_task_end. Both unitrace (with --chrome-itt-logging) and VTune (Tasks tab) will show layer‑named regions framing the kernels — getting you most of PyTorch profiler's module attribution without the framework hooks.

Cross‑tool gotchas on Intel Xe / Xe2

GotchaAffectsFix
Forgetting torch.xpu.synchronize() before __itt_pause() VTune · unitrace Empty/clipped GPU timeline. Always sync before pausing.
torch.compile + SYCL JIT → llvm-foreach SEGV or unreadable names all three Launch with --enforce-eager on intel/vllm-xpu.
Manually setting ZE_ENABLE_TRACING_LAYER=1 VTune Conflicts with VTune's own tracer → empty GPU timeline. Leave it unset.
vLLM v1 spawns EngineCore in a subprocess; ITT in API server doesn't reach the worker VTune · unitrace Patch vllm.v1.worker.gpu_worker.Worker.execute_model; or use a filesystem gate file the worker re‑checks each step.
BMG/Xe2 + recent oneAPI: ze_ipc_event_counter_based_handle_t cast error in pti‑gpu unitrace build Patch scripts/gen_tracing_callbacks.py::gen_to_hex_string_functions with the templated helper, regenerate, rebuild.
BMG/Xe2 (xe kernel driver) + VTune 2025.x: “Hottest GPU Computing Tasks” comes back empty VTune GPU‑Hotspots & gpu‑offload VTune 2025.x metric‑discovery was validated on i915. Collection runs but per‑kernel timing is missing. Use unitrace for BMG kernel attribution until VTune 2026+ ships xe‑native support. run_vtune_vllm.sh WARNs at preflight when it detects xe.
VTune GPU‑Hotspots “Cannot collect GPU hardware metrics · neither libigdmd.so nor libmd.so was found” VTune Install intel-metrics-discovery and, on Ubuntu 24.04, create the missing unversioned symlink: ln -sf libigdmd.so.1 /usr/lib/x86_64-linux-gnu/libigdmd.so && ldconfig.
Profiling without warmup all three Always run a few warmup requests before /start_profile; otherwise compile, autotune, and KV alloc dominate the trace.
Treating "more decode steps captured" as "decode got slower" any timeline tool Normalize by step (per‑step kernel time), not by absolute totals. Prefill steps and decode steps land in the trace at the rate the ROI window allows.

One‑line summary

PyTorch profiler tells you which layer. unitrace tells you where on the timeline. VTune tells you why on the silicon. Pick the question first; the tool follows.
