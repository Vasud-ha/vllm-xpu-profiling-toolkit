vLLM PyTorch Profiler + Perfetto Analysis

Vendor-agnostic vLLM profiling using the official --profiler-config CLI arg (vLLM ≥ 0.17) or the legacy VLLM_TORCH_PROFILER_DIR env var (≤ 0.14), the built-in start_profile / stop_profile endpoints, and Perfetto / chrome-tracing for analysis. Works on NVIDIA CUDA and Intel XPU builds.
torch.profiler vLLM v1 (0.14 / 0.17 / 0.21) Perfetto / chrome-tracing Vendor-agnostic ROI = warmup-then-tight-window

Authoritative source: docs.vllm.ai — profiling index

1. Summary

vLLM exposes torch.profiler behind three layers — a version-dependent server gate, programmatic start_profile / stop_profile, and bench-driven vllm bench serve --profile. Traces are .pt.trace.json files (.json.gz when torch_profiler_use_gzip is set) opened in Perfetto (ui.perfetto.dev).

vLLM 0.17 changed the gate
On vLLM ≥ 0.17 the VLLM_TORCH_PROFILER_DIR env var is silently ignored — the server logs "Unknown vLLM environment variable detected: VLLM_TORCH_PROFILER_DIR" and curl -X POST /start_profile returns 404 because the profile router only attaches when profiler_config.profiler is not None. Pass a ProfilerConfig JSON via --profiler-config instead. run_pt_profile_vllm.sh sets both — env var and CLI arg — so it works on any supported release.
Goal
Capture a tight, warm-state window — 1–2 prefill steps + 5–20 decode steps — so the trace stays in the hundreds of MB and Perfetto can render it without choking.

2. Technical Approach

2.1 The three layers

LayerEntry pointWhen to use
CLI-gated server (vLLM ≥ 0.17) --profiler-config '{"profiler":"torch","torch_profiler_dir":"/abs/path"}' + POST /start_profile / /stop_profile Current path — required for the profile router to register on 0.17+
Env-var gated server (vLLM ≤ 0.14) VLLM_TORCH_PROFILER_DIR=/path + POST /start_profile / /stop_profile Legacy path — silently no-ops on ≥ 0.17
Programmatic LLM.start_profile() / stop_profile() (offline) or AsyncLLMEngine.start_profile() / stop_profile() (async) Custom drivers / sidecar scripts
Bench-driven vllm bench serve --profile (server started with one of the gates above) Reproducible end-to-end captures

2.2 Capture flow

  1. Pick the gate: --profiler-config on ≥ 0.17, or VLLM_TORCH_PROFILER_DIR on ≤ 0.14. The wrapper sets both.
  2. Start vllm serve … with --enforce-eager (esp. on Intel XPU).
  3. Warm up outside the ROI — first call triggers CUDA graph capture, Triton/torch.compile autotune, KV alloc.
  4. POST /start_profile (or use vllm bench serve --profile).
  5. Drive a tight workload: --num-prompts 4–16, short outputs.
  6. POST /stop_profile — flush happens here.
  7. Open the *.pt.trace.json in Perfetto.
Single most common failure
On 0.17+, forgetting --profiler-config — the endpoint 404s. On ≤ 0.14, VLLM_TORCH_PROFILER_DIR was in the user's shell but not in the server's environment (Docker missing the env-pass). Endpoints exist but silently do nothing.

3. Commands & APIs

3.1 Gate (pick by vLLM version)

# vLLM ≥ 0.17 — pass a ProfilerConfig JSON on the CLI
TRACE_DIR="$HOME/vllm_profile"
mkdir -p "$TRACE_DIR"

vllm serve meta-llama/Llama-3.1-8B-Instruct \
  --enforce-eager \
  --profiler-config "{\"profiler\":\"torch\",\"torch_profiler_dir\":\"$TRACE_DIR\",\"torch_profiler_use_gzip\":true}"

# vLLM ≤ 0.14 — legacy env-var gate (silently ignored on ≥ 0.17)
export VLLM_TORCH_PROFILER_DIR="$TRACE_DIR"
mkdir -p "$VLLM_TORCH_PROFILER_DIR"
vllm serve meta-llama/Llama-3.1-8B-Instruct --enforce-eager

The gate must be in place before the engine starts; the trace dir must be writable by the server uid.

3.2 Server + bench (recommended)

# Terminal 1 — server with the 0.17+ CLI gate
export TRACE_DIR="$HOME/vllm_profile"
mkdir -p "$TRACE_DIR"
vllm serve meta-llama/Llama-3.1-8B-Instruct \
  --enforce-eager \
  --max-num-seqs 32 \
  --max-num-batched-tokens 8192 \
  --gpu-memory-utilization 0.85 \
  --profiler-config "{\"profiler\":\"torch\",\"torch_profiler_dir\":\"$TRACE_DIR\",\"torch_profiler_use_gzip\":true}"

# Terminal 2 — drives /start_profile and /stop_profile around the run
vllm bench serve \
  --backend openai \
  --model meta-llama/Llama-3.1-8B-Instruct \
  --dataset-name random \
  --random-input-len 1024 \
  --random-output-len 128 \
  --num-prompts 8 \
  --request-rate inf \
  --profile

3.3 HTTP endpoints directly

curl -X POST http://localhost:8000/start_profile
# ... drive traffic ...
curl -X POST http://localhost:8000/stop_profile

3.4 Offline LLM API

# vLLM ≥ 0.17 — construct a ProfilerConfig
from vllm import LLM, SamplingParams
from vllm.config import ProfilerConfig   # available on 0.17+

llm = LLM(
    model="meta-llama/Llama-3.1-8B-Instruct",
    enforce_eager=True,
    profiler_config=ProfilerConfig(
        profiler="torch",
        torch_profiler_dir="/abs/path",
        torch_profiler_use_gzip=True,
    ),
)

# vLLM ≤ 0.14 — env var still honored offline
# os.environ["VLLM_TORCH_PROFILER_DIR"] = "/abs/path"
# llm = LLM(model="meta-llama/Llama-3.1-8B-Instruct", enforce_eager=True)

# Warmup OUTSIDE the profile window
llm.generate(["warmup"] * 4, SamplingParams(max_tokens=8))

llm.start_profile()
out = llm.generate(prompts, SamplingParams(max_tokens=64))
llm.stop_profile()

3.5 Async engine

from vllm import AsyncEngineArgs, AsyncLLMEngine
engine = AsyncLLMEngine.from_engine_args(AsyncEngineArgs(model=...))

await _drive_warmup(engine)
await engine.start_profile()
await _drive_roi(engine)
await engine.stop_profile()

4. Scripts & Recipes

The toolkit ships three files under pytorch-profiler/scripts/:

FileRole
scripts/run_pt_profile_vllm.sh Launcher — preflight checks, launches the wrapper with the 0.17+ --profiler-config flag and the ≤ 0.14 VLLM_TORCH_PROFILER_DIR env var, warmup, ROI bench, stop_profile flush, trace inventory, summary.
scripts/serve_with_pt_profile.py vLLM wrapper that patches vllm.v1.worker.gpu_worker.Worker.execute_model with a torch.profiler.record_function("vllm.<phase> bs=… tok=…") span. Adds structured labels around vLLM's default annotations; does not drive torch.profiler itself — vLLM's built-in profiler owns that.
scripts/summarize_trace.py Post-run quantitative summary: step cadence, forward fraction, top kernels by device time, phase mix.

4.1 Environment baked into the launcher

HF_HOME=/hf_cache
TORCH_LLM_ALLREDUCE=1
VLLM_ALLOW_LONG_MAX_MODEL_LEN=1
VLLM_WORKER_MULTIPROC_METHOD=spawn
VLLM_ENGINE_READY_TIMEOUT_S=1800     # raised for vLLM 0.21 cold weight-load on NFS

The launcher unsets ZE_ENABLE_TRACING_LAYER / ZE_LOADER_LAYERS_ENABLE / PTI_ENABLE_COLLECTION / PTI_ENABLE_RUNTIME_TRACING — Level-Zero tracing layers left over from a VTune-style run break XPU activity capture.

4.2 Phase-isolated captures

# Prefill-heavy (long prompt, single-token completion)
PT_PHASE=prefill INPUT_LEN=2048 OUTPUT_LEN=1 NUM_PROMPTS=8 \
  bash run_pt_profile_vllm.sh

# Decode-heavy (short prompt, long completion)
PT_PHASE=decode INPUT_LEN=128 OUTPUT_LEN=512 NUM_PROMPTS=8 \
  bash run_pt_profile_vllm.sh

# Both (default) — mixed steps get the mixed label
PT_PHASE=both bash run_pt_profile_vllm.sh

PT_LABEL_EVERY_STEP=0 only labels phase transitions instead of every step (cuts label spam in long captures).

4.3 Keeping traces small

  1. Tight ROI — 1–2 prefill steps + 5–20 decode steps. Don't profile a whole bench run.
  2. Small batch / low concurrency — --num-prompts 4–8, short outputs.
  3. --enforce-eager — disables CUDA graphs so kernel names are real, and avoids capturing graph-build noise.
  4. gzip the traces — set torch_profiler_use_gzip=true in the ProfilerConfig; Perfetto reads .pt.trace.json.gz directly.
  5. Avoid record_shapes / with_stack / profile_memory unless explicitly chasing those questions — they multiply trace size.
  6. TP: each rank writes its own .pt.trace.json; open them separately first, merge only when needed.

4.4 Annotate your own ROIs (no fork required)

import torch
with torch.profiler.record_function("my_roi"):
    # whatever you want labeled in Perfetto
    ...

4.5 Cheap pre-checks before opening a multi-hundred-MB file

ls -lh "$TRACE_DIR"

# Peek at JSON header
python -c "
import sys
with open(sys.argv[1]) as f:
    print(f.read(2000))
" "$TRACE_DIR"/*.pt.trace.json | head -c 2000

# For .json.gz:
zcat "$TRACE_DIR"/*.pt.trace.json.gz | head -c 2000

Look for the traceEvents array, process_name / thread_name metadata, and recognizable kernel names (flash_attn, paged_attn, gemm, all_reduce, xetla on Intel).

5. Perfetto Analysis

5.1 Lane layout to expect

  1. API server / asyncio — short request-handling spans (online server only).
  2. Engine / scheduler — Scheduler.schedule, EngineCore.step.
  3. Worker process(es) — one per TP rank; Worker.execute_model blocks per step, wrapped by the vllm.<phase> bs=… tok=… spans emitted by serve_with_pt_profile.py.
  4. CUDA / XPU streams — actual kernel timeline.
  5. NCCL / oneCCL streams (TP > 1) — collective comms.

5.2 Identifying prefill vs. decode

  • Prefill: one-shot, large; a single execute_model block far wider than its neighbors. Attention time scales with batch_seq_len² (or batch_seq_len × head_dim for FlashAttention).
  • Decode: many, small, near-uniform; cadence ~2–15 ms/step depending on hw. Sampling/logits blocks are visible and non-trivial.

5.3 Quantitative starting points

MetricHealthy rangeOut-of-range means
Step cadence (steady-state decode)5–12 ms (8B fp16, modern GPU); higher on Intel BMGCompare to model size + hw
Forward fraction (forward kernel time / step)> 0.85 in pure decode< 0.7 → CPU/scheduler is the bottleneck
Attention fraction within forward0.5–0.8 long-context decode; 0.2–0.4 short-contextBackend / config issue
NCCL / oneCCL fraction (TP only)< 0.2 steady-state> 0.2 → TP overprovisioned for batch

5.4 Useful Perfetto features

  • Pivot Table (Query → "Slice ↦ Stats by name") — total time per kernel name.
  • SQL on trace_processor — for traces too big to scroll: 
    SELECT name, COUNT(*) AS n, SUM(dur)/1e6 AS total_ms
FROM slice WHERE category = 'kernel'
GROUP BY name ORDER BY total_ms DESC LIMIT 20;
  • Flow events — connect a Python op on CPU to its dispatched kernel on GPU.
  • Pin lanes — pin worker stream + NCCL/CCL stream side-by-side for comm/compute alignment.

6. Hotspot Patterns & Pathologies

6.1 What healthy hotspots look like

  • Prefill: GEMMs (QKV/FFN/LM head) and attention kernel dominate. flash_attn_*, paged-attention prefill paths.
  • Decode: paged-attention decode, small-K GEMMs, all_reduce on TP, sampling kernels (top_k, softmax, argmax), norm.

6.2 Pathology table

7. Optimization — Trace findings → vLLM knobs

Trace patternKnob to tryNotes
GPU idle gaps in decode--max-num-seqs ↑, --max-num-batched-tokens ↑, --enable-chunked-prefillDefault v1 has chunked prefill on
Allocator churn / KV fragmentation--block-size 16 → 32, --gpu-memory-utilization 0.90 → 0.85Larger blocks = less metadata, more wasted slots when seqs end
Attention dominates decodeVLLM_ATTENTION_BACKEND=FLASH_ATTN / FLASHINFER (CUDA); auto-selected on XPURe-capture and compare attention-fraction
Memory-bound decode--kv-cache-dtype fp8 / int8KV-cache quant matters more than weight quant for decode
NCCL / oneCCL fraction > 0.2Lower --tensor-parallel-size or raise concurrency / --max-num-seqsComm cost is fixed — amortize
Spec decode underperformingSmaller draft, lower --num-speculative-tokensAcceptance > 50 % is the bar

Order of changes (one knob per capture)

  1. Engine version + --enforce-eager for analysis sanity.
  2. Batching (--max-num-seqs, --max-num-batched-tokens, chunked prefill).
  3. Memory (--gpu-memory-utilization, --block-size).
  4. Attention backend.
  5. Quantization (weights, then KV cache).
  6. Parallelism (TP / PP).
  7. Speculative decoding.

Capture between steps. Only end-to-end numbers matter.

Intel XPU caveat
Quantization-related slowdowns may not be kernel issues — on bmg3 the int8 XMX kernel was fine but the GPU hit Burst Power Excursion, causing ~24 % freq drop. Cross-check power/thermal counters before blaming the kernel.

8. Troubleshooting

8.1 Trace dir empty after stop_profile

  1. vLLM ≥ 0.17: confirm --profiler-config was passed to vllm serve. The wrapper logs Using --profiler-config CLI arg (vLLM >= 0.17 gate) when it detects the flag; if you see Neither VLLM_TORCH_PROFILER_DIR nor --profiler-config is set, the gate is missing.
  2. vLLM ≤ 0.14: verify VLLM_TORCH_PROFILER_DIR is in the server's environment (not just your shell). Containers often miss it.
  3. Dir exists and server uid can write (ls -ld $TRACE_DIR).
  4. start_profile returned 200 (curl -i).
  5. Did you call stop_profile? Trace flushes on stop.
  6. Process exited cleanly. SIGKILL mid-profile loses the buffer — use SIGTERM.

8.2 GPU lane is empty

CUDA: CUPTI not loadable. Verify with python -c "import torch; torch.cuda.profiler.start(); torch.cuda.profiler.stop()". Locked-down hosts may need RmProfilingAdminOnly relaxed. Inside Docker: --cap-add=SYS_ADMIN + CUPTI on LD_LIBRARY_PATH.

Intel XPU: torch / IPEX version mismatch silently produces CPU-only traces. Always launch with --enforce-eager on intel/vllm-xpu. Verify with a standalone check:

import torch
from torch.profiler import profile, ProfilerActivity
x = torch.randn(1024, 1024, device="xpu")
with profile(activities=[ProfilerActivity.CPU, ProfilerActivity.XPU]) as p:
    (x @ x).sum().item()
print(p.key_averages().table(sort_by="xpu_time_total", row_limit=5))

If this shows zero XPU rows, the platform side is broken — fix before involving vLLM.

9. Conclusion

Key takeaways
  1. Pick the gate by version: --profiler-config on vLLM ≥ 0.17; VLLM_TORCH_PROFILER_DIR on ≤ 0.14. The env var is silently ignored on 0.17+. Setting both is the safe default for cross-version scripts.
  2. Always warm up before start_profile. CUDA-graph capture / Triton autotune / KV alloc must be out of the ROI or steady-state numbers are wrong.
  3. Keep traces in the hundreds of MB: tight ROI, small batch, short outputs, --enforce-eager, gzip.
  4. Verify GPU lane is non-empty before deep analysis. CUPTI (CUDA) / Level Zero + XPU (Intel) issues silently produce CPU-only traces.
  5. Use Perfetto's pivot table or trace_processor SQL for kernel ranking; eyeballing is too slow for production traces.
  6. Map findings to one knob at a time, in the order: engine → batching → memory → attention → quant → parallelism → spec decode.

9.1 Cross-skill notes

  • For Intel Xe (BMG / PVC) + ITT-driven ROI control with VTune GPU-Hotspots, see the companion vtune skill. VTune produces .vtune result-dirs; PyTorch profiler produces Perfetto-readable .pt.trace.json.
  • For unitrace (pti-gpu) on Intel Xe, see the unitrace skill — same ITT ROI pattern, different collector, per-process Chrome-trace JSON.
