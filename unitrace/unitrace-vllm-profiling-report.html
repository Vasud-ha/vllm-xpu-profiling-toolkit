unitrace + vLLM Conditional Profiling on Intel Xe

Build, install, and run Intel pti-gpu unitrace against vLLM serving workloads on Intel Xe / Xe2 (BMG) GPUs, with ITT-driven /start_profile and /stop_profile endpoints so any HTTP client can bracket the ROI.
Intel pti-gpu unitrace vLLM v1 Intel Xe / Xe2 (BMG) Level Zero + SYCL + ITT Chrome trace JSON --start-paused + ITT resume/pause

Authoritative source: github.com/intel/pti-gpu/tree/master/tools/unitrace — upstream README, build instructions, and option reference.

1. Summary

unitrace (intel/pti-gpu) hooks Level Zero (and optionally OpenCL / SYCL / oneCCL / oneDNN) at runtime and writes a Chrome-trace JSON per process, plus an aggregated device-timing summary at exit. Capturing the entire vLLM server lifetime is wasteful (model load, KV alloc, warmup, idle ticks all land in the trace), so this skill drives a conditional ROI:

  • Launch with unitrace --start-paused.
  • Replace vLLM's torch-profiler-backed /start_profile + /stop_profile with ITT-driven equivalents (__itt_resume / __itt_pause) plus a filesystem gate file for cross-process glue.
  • Patch vllm.v1.worker.gpu_worker.Worker.execute_model so the worker subprocess re-checks the gate file every step and toggles its own ITT state — belt-and-suspenders.

Validated on gnrsp-bmg3.iind.intel.com against:

  • intel/vllm:0.14.1-xpu (vllm 0.1.dev14456+gde3f7fe65) — original validation environment.
  • intel/vllm:0.17.0-xpu — re-validated in container unitrace_vllm_test_vasu; the ROI flow, ITT symbol resolution, and Chrome-trace output behave identically. No script changes were required.
  • intel/vllm:0.21.0-ubuntu24.04 — re-validated in container skill-test-vasu; see verification_log_vllm_0.21.md in the toolkit repo. --metric-query path (ComputeBasic, DeviceCacheProfile, VectorEngineStalls, etc.) works on BMG here; only --stall-sampling still doesn't propagate into the spawned EngineCore.

Validation base: oneAPI 2025.3, kernel 6.17, Intel Graphics [0xe223] (BMG / Xe2), unitrace 2.3.0.

2. Technical Approach

2.1 Why ROI matters

Without ROI gating, the trace covers the whole server lifetime — load_format, KV-cache allocation, warmup, chat-template processing, idle scheduler ticks. With --start-paused + ITT calls, unitrace records only the inference window — the same control mechanism VTune uses.

2.2 ROI control contract

client (curl / OpenAI SDK / vllm bench serve) | | POST /start_profile v FastAPI handler ──► __itt_resume() ──► unitrace begins capture | + create gate file | | ... inference traffic ... | | POST /stop_profile v FastAPI handler ──► torch.xpu.synchronize() ──► __itt_pause() ──► unitrace stops capture ──► remove gate file

2.3 Why the gate file

vLLM v1 spawns the EngineCore in a subprocess; ITT calls in the API-server process don't reach the worker. The gate file is the cross-process glue: a patched Worker.execute_model in the worker process re-checks it every step and toggles its own __itt_resume / __itt_pause around the kernel launches.

2.4 ITT symbol resolution

The intel/vllm:0.14.1-xpu container has no libittnotify.so — only the static .a from pti-gpu. unitrace itself compiles ittnotify_static.c into libunitrace_tool.so and forwards __itt_resume / __itt_pause from there. The wrapper resolves symbols in this order:

  1. RTLD_DEFAULT — unitrace already injected via LD_PRELOAD.
  2. libunitrace_tool.so by absolute path with RTLD_GLOBAL (covers running outside unitrace).
  3. find_library("ittnotify") — last-resort fallback for VTune installs.
$ nm -D libunitrace_tool.so | grep -E '__itt_resume|__itt_pause'
00000000000fc940 T __itt_pause
00000000000fcaf0 T __itt_pause_scoped
00000000000fcb00 T __itt_resume
00000000000fccb0 T __itt_resume_scoped
Important
Always combine --start-paused with the ITT calls. Without --start-paused, ITT pause/resume just toggles whether the existing collection is "active" — it does not reduce the captured-data window for the whole trace.

3. Build (Linux, BMG / Xe2)

Requires CMake ≥ 3.22, C++17, oneAPI Base Toolkit, Python ≥ 3.9.

git clone --depth 1 https://github.com/intel/pti-gpu.git
cd pti-gpu/tools/unitrace
mkdir build && cd build

source /opt/intel/oneapi/setvars.sh --force

cmake -DCMAKE_BUILD_TYPE=Release \
      -DBUILD_WITH_MPI=0 \
      -DBUILD_WITH_ITT=1 \
      -DBUILD_WITH_XPTI=1 \
      -DBUILD_WITH_OPENCL=1 \
      ..
make -j$(nproc)

Artifacts:

  • build/unitrace — launcher binary.
  • build/libunitrace_tool.so — injected into the target process; must stay alongside the binary.
  • build/ittapi/bin/libittnotify.a — static ITT lib unitrace links against.

3.1 BMG/Xe2 build break — (uintptr_t)(val) cast on struct handle

Newer Level Zero headers (oneAPI 2025.3 / recent compute-runtime) make ze_ipc_event_counter_based_handle_t a struct, but pti-gpu's generator emits a C-style (uintptr_t)(val) cast inside TO_HEX_STRING that fails:

tracing.gen: error: invalid cast from type 'ze_ipc_event_counter_based_handle_t'
{aka '_ze_ipc_event_counter_based_handle_t'} to type 'uintptr_t'
  7 |     std::sprintf(buffer, "0x%lx", (uintptr_t)(val)); \
    |                                   ^~~~~~~~~~~~~~~~

Fix — patch scripts/gen_tracing_callbacks.py::gen_to_hex_string_functions to emit a templated helper that dispatches on type traits:

f.write("#include <type_traits>\n")
f.write("template <typename T>\n")
f.write("static inline uintptr_t to_hex_value_(const T& v) {\n")
f.write("  if constexpr (std::is_pointer_v<T>) {\n")
f.write("    return reinterpret_cast<uintptr_t>(v);\n")
f.write("  } else if constexpr (std::is_integral_v<T> || std::is_enum_v<T>) {\n")
f.write("    return static_cast<uintptr_t>(v);\n")
f.write("  } else {\n")
f.write("    uintptr_t out = 0;\n")
f.write("    std::memcpy(&out, &v, sizeof(v) < sizeof(out) ? sizeof(v) : sizeof(out));\n")
f.write("    return out;\n")
f.write("  }\n")
f.write("}\n")
f.write("#define TO_HEX_STRING(str, val) \\\n")
f.write("    {char buffer[32]; \\\n")
f.write("    std::sprintf(buffer, \"0x%lx\", (unsigned long)to_hex_value_(val)); \\\n")
f.write("    str += std::string(buffer); \\\n")
f.write("    }\n")

Then regenerate and rebuild:

cd build
rm -f tracing.gen common_header.gen l0_loader.gen
make -j$(nproc)

4. Launch & ROI Commands

4.1 Two-file launcher

  • scripts/serve_with_unitrace.py — vLLM wrapper with ITT-driven endpoints.
  • scripts/run_unitrace_vllm.sh — unitrace --start-paused entrypoint, with an EXIT trap that auto-runs generate_report.py against $RESULT_DIR.

4.2 Default unitrace flags

--chrome-kernel-logging     # GPU kernels in Chrome trace
--chrome-sycl-logging       # SYCL runtime calls

4.5 Examples

# Tight GPU-only trace (fast, small)
UNITRACE_PRESET=lite bash run_unitrace_vllm.sh

# Multi-rank/TP run with allreduce visibility
TP_SIZE=2 UNITRACE_PRESET=ccl bash run_unitrace_vllm.sh

# Hunting host overhead — show every L0 dispatch call
UNITRACE_PRESET=call bash run_unitrace_vllm.sh

# Verbose + 1M-event buffer cap
UNITRACE_EXTRA="--verbose --chrome-event-buffer-size 1000000" bash run_unitrace_vllm.sh

# Smaller traces by dropping per-thread / per-engine subdivision
UNITRACE_EXTRA="--chrome-no-thread-on-device --chrome-no-engine-on-device" \
  bash run_unitrace_vllm.sh

# Show me everything (big traces — keep ROI short)
UNITRACE_PRESET=full bash run_unitrace_vllm.sh

# Manual override — full control, ignores preset
UNITRACE_FLAGS="--start-paused -d --chrome-kernel-logging --chrome-call-logging" \
  bash run_unitrace_vllm.sh

4.6 Driving the ROI

PORT=9090

# 1. Start ROI
curl -X POST http://localhost:$PORT/start_profile

# 2. Issue inference — any client works
curl -X POST http://localhost:$PORT/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"meta-llama/Llama-3.1-8B-Instruct",
       "messages":[{"role":"user","content":"San Francisco is a"}],
       "max_tokens":24}'

# 3. Stop ROI
curl -X POST http://localhost:$PORT/stop_profile

# 4. Stop the server (Ctrl-C). unitrace flushes the trace and prints
#    the device-timing summary to stdout. The launcher's EXIT trap
#    auto-runs generate_report.py against $RESULT_DIR.
Don't use -- separator
unitrace -- python … rejects the --. Just unitrace <flags> python ….

6.4 Quick health check

python3 - <<'PY'
import json, collections, glob, os
d = sorted(glob.glob("unitrace_results/*/"))[-1]
for p in glob.glob(os.path.join(d, "python.*.json")):
    data = json.load(open(p))
    ev = data.get("traceEvents", data) if isinstance(data, dict) else data
    cats = collections.Counter(e.get("cat","-") for e in ev)
    gpu = [e for e in ev if e.get("cat")=="gpu_op" and e.get("ph")=="X"]
    span = 0
    if gpu:
        ts = sorted(e["ts"] for e in gpu if "ts" in e)
        span = (ts[-1]-ts[0])/1e6 if ts else 0
    print(f"{os.path.basename(p)}: {len(ev)} events, gpu={len(gpu)}, span={span:.2f}s")
PY

Healthy output for a single short chat completion (tight ROI):

python.7954.json: 200 events,  gpu=0,    span=0.00s   # API server, no kernels
python.8236.json: 29064 events, gpu=1164, span=0.48s  # EngineCore — kernels in tight window

6.5 What to look at first

  1. Where is GPU time spent? Read the device-timing summary — top kernels by % time tell you what to optimize.
  2. Are kernels back-to-back, or are there gaps? Open the EngineCore JSON in Perfetto and zoom in. Visible gaps = host-bound overhead (sampling, scheduling, sync).
  3. Unexpected H2D / D2H copies? Steady-state decode should have almost none.
  4. Per-iteration shape — find one full execute_model cycle (between two ITT resume markers if --chrome-itt-logging is on); that kernel sequence is your inner loop.

6.7 Trace size warnings

--chrome-kernel-logging + --chrome-sycl-logging for a 30 s benchmark of a 7B model can hit hundreds of MB. Chrome refuses traces over ~256 MB; Perfetto handles > 1 GB but slowly. Mitigations: tighten ROI, drop --chrome-sycl-logging if only kernel timing matters, use --include-kernels=<substr-list> (Level Zero only) to filter.

Trace files arrive AFTER the process exits
unitrace writes JSONs during teardown. kill -9 the launcher → truncated/missing JSONs. Always Ctrl-C (one is enough; give a few seconds to flush). The device-timing summary printing to stdout is the sign that teardown completed.

7.1 Verifying ITT is wired up

The launcher prints:

2026-06-10 12:15:39 [unitrace.vllm] ITT available: True

You want True in both the API-server process AND the EngineCore_DP0 subprocess. If only one says True, the worker process didn't inherit the injected lib.

The struct-cast patch is not yet upstream. After git pull, re-apply it or check the upstream intel/pti-gpu repo.

BMG3 takeaway
  1. For per-kernel GPU performance counters on BMG3, none of the metric-query / metric-sampling / stall-sampling paths work cleanly under vLLM v1. Use VTune GPU-Hotspots (see the companion vtune-vllm-profiling skill) instead.
  2. Chrome-trace logging, host/device timing tables, kernel filters, ROI control (--start-paused + ITT + --session) all work end-to-end.
  3. The deprecated --conditional-collection alias still works silently — but prefer --start-paused for clarity.

9. Conclusion

Key takeaways
  1. unitrace + ITT + --start-paused ≡ tight ROI. Without all three, you're back to a whole-lifetime trace.
  2. Symbol resolution model: ITT comes from libunitrace_tool.so, not a system libittnotify.so — the container doesn't ship one.
  3. Cross-process glue: the gate file is essential for v1 — ITT calls in the API parent don't reach the EngineCore subprocess.
  4. BMG/Xe2 build patch: apply the templated to_hex_value_ helper to gen_tracing_callbacks.py until upstream merges a fix.
  5. Always Ctrl-C, never SIGKILL. unitrace flushes JSONs at teardown; kill -9 truncates them.
  6. Pick the smallest preset that answers your question. lite for first pass, call/ccl/dnn for targeted investigations, full only when you need everything.
